Section 02 · 续

LLM 直接代码生成能力评测

LLM生成SVG本质是代码生成任务，SVG是代码。这是与IDE场景的SVG Skill最直接竞争的方向。

核心洞察：SVG 支持注释，LLMs 几乎都会在生成的代码中加入注释，这让我们能看出它们的"想法"和意图——即使视觉结果很糟糕，注释也暴露了模型的空间推理逻辑。

Section 04

Skill / Agent 场景空白分析

目前市面上没有专门为 AI 编程 IDE 设计的 SVG 插画生成 Skill，存在四大核心空白。

现状：现有工具均需切换应用，无一针对开发者编码工作流设计，缺乏项目上下文感知能力，这是最大的产品机会所在。

① 开发者工作流集成
开发者在写前端代码时直接需要 SVG 素材，所有现有工具均需切换应用，严重打断编码流。没有任何工具能在 IDE 内直接生成 SVG 并插入代码。
② 代码上下文感知
生成的 SVG 需要符合项目的颜色变量（CSS Custom Properties）、设计系统（Tailwind Token）。现有所有工具均不理解代码上下文，生成结果与项目设计规范往往不一致。
③ 迭代修改能力
能够基于现有 SVG 进行语义编辑（如"把左边的圆改成星形"），保持多轮对话上下文。Claude 在这方面最强，但没有工具把这个能力包装成开发者友好的 Skill。
④ 插画 vs 图标的生态差距
市面工具多针对图标场景（单色、简单几何），而复杂插画场景（空状态页、引导页、营销插画、人物场景）几乎无专门支持，是最大的内容空白。
Section 05

主要挑战

构建高质量 SVG 插画生成 Skill 需要正视技术和评估两类挑战。

技术挑战

空间推理弱 高风险
LLM 难以精确控制形状位置和比例，多元素场景易错乱，超过 5 个元素质量急剧下降
复杂场景 高风险
多元素相对关系难以把控，人物、场景等复杂插画几乎不可用，需要有效的复杂度控制策略
路径优化 中风险
LLM 生成的 SVG 代码往往冗余，文件体积大，需要 SVGO 等工具后处理
视觉一致性 中风险
风格统一难度大，多次生成结果差异大，需要风格约束机制
渐变 / 动画 中风险
复杂视觉效果支持有限，模型容易生成错误语法，需要校验层

评估挑战

SVG 质量评估困难，传统像素级指标（MSE、SSIM）与人类感知相关性弱，需要专门的评估体系。
学术方案
VectorGym（2025）提出 VLM-as-Judge 方案，经人类评测验证有效。使用 Gemini 3 Pro 作为评判模型，Pearson 相关系数达到最高水平。
实践指标体系
• 视觉相似度：LPIPS + DINO 特征 • 语义对齐：CLIP 相似度 • 代码质量：语法有效率 + 路径复杂度 • 人工评测：风格一致性 + 可编辑性

建议：建立内部 SVG 质量评测集，覆盖图标、插画、装饰元素三类场景，用于指导模型选型和 Prompt 优化。

Section 06

机会分析：SVG插画生成 Skill 定位

基于调研，我们的 Skill 应占据「高复杂度 × 开发者工具」象限，以上下文感知为核心差异化。

Core Positioning
面向开发者的上下文感知 SVG 插画生成助手
在 IDE 内直接生成，读懂你的代码，输出符合项目设计系统的可用 SVG

五大差异化能力

🎨 项目色彩感知
读取 CSS 变量 / Tailwind Token，用项目的颜色体系生成 SVG，告别颜色不统一的烦恼
CSS VariablesTailwind
📚 风格库管理
记住用户偏好的插画风格（flat、线稿、等距、卡通等），一键切换风格，保持项目一致性
Flat Design等距插画线稿
🔄 增量迭代修改
基于已有 SVG 进行语义编辑，支持自然语言描述修改意图，保持多轮对话上下文
SVG Editing多轮对话
⚛️ 组件感知输出
理解 React / Vue 组件结构，直接输出带 Props 接口的 SVG 组件，而不只是裸 SVG 代码
ReactVueTypeScript
⚡ 优化输出
自动进行路径压缩（SVGO），确保代码简洁、加载快，同时进行 SVG 语法校验，禁止无效属性
SVGO路径简化语法校验

  • 图标集合 — 项目定制图标
  • 装饰元素 — 背景、分割线、点缀

Section 08

结论与建议

综合市场现状、技术成熟度和空白分析，给出产品决策建议。

维度结论
市场时机 ✅ 好时机 LLM SVG生成能力已成熟，但 IDE 集成 Skill 空白，是入场最佳窗口
底座模型推荐 Claude Sonnet 4.5 或 Gemini 3 Pro（VectorGym 评测中双双领先，两者各有侧重）
优先场景 简单到中等复杂度插画 — 空状态页、404页、装饰元素、引导插画（前端高频需求）
MVP 核心功能 文本描述 → SVG 代码 + 风格引导 + SVGO 优化输出
壁垒建设方向 积累插画风格 Prompt 库、建立 SVG 组件生态、沉淀用户风格偏好数据
短期主要风险 ⚠️ 注意 LLM 空间推理仍有明显局限，5个元素以上复杂场景效果不稳定

三阶段产品路线

Phase 1 · MVP
文本→简单插画
聚焦前端高频场景：空状态页、404页、装饰元素。

底座：Claude Sonnet 4.5
输出：带 SVGO 优化的 SVG 代码
关键：建立 Prompt 模板库，覆盖10+常见风格
空状态页404页装饰元素
Phase 2 · 差异化
上下文感知集成
读取项目颜色变量和组件规范，输出符合设计系统的 SVG 组件。

功能：CSS Variable 读取 / Tailwind Token 解析
输出：React / Vue 组件形式
关键：这是与所有现有工具的核心差异点
CSS变量设计系统组件输出
Phase 3 · 壁垒
专用模型微调
引入 RL 微调的专用小模型，参考 Qwen3VL 8B Gym 方案，提升生成质量同时降低调用成本。

方案：RLRF + 渲染反馈奖励
目标：开源小模型质量超越 Claude Sonnet
关键：积累内部 SVG 评测集 + 训练数据
RL微调RLRF专用模型

最终判断：现在是入场 SVG 插画生成 Skill 的最佳时机——LLM 能力已足够实用，专业工具无法集成进 IDE，开发者工作流空白巨大。以「上下文感知」为核心差异，以「Claude/Gemini 为底座 + RL 微调为壁垒」为技术路径，分三阶段稳步推进。
